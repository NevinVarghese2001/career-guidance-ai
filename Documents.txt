**Product Note – Career Guidance AI**

## Problem Statement

Students completing secondary education (10th / 12th) in Kerala often struggle to choose the right academic or career path. Decisions are frequently influenced by peer pressure, limited awareness of alternatives, financial constraints, and lack of access to personalized counseling. Existing guidance is either generic or inaccessible, leading to poor course choices and dissatisfaction later.

The problem chosen for this project is to provide **structured, personalized, and explainable career guidance** to students at this critical decision stage.

---

## Intended User

The primary users of this system are:

* Students who have completed 10th or 12th grade
* First-generation learners
* Students from budget-constrained families
* Students who prefer local (Kerala-based) education options

Secondary users include career counselors who can use the system as a **decision-support tool**.

---

## Key Design Decisions

### 1. Structured Question-Based Input

Instead of free-form inputs, the system collects structured responses related to academic background, interests, personality, constraints, and career goals. This ensures consistency and allows meaningful analysis.

### 2. Offline Machine Learning Model

A **Decision Tree classifier** was chosen as the core AI component. This model works well with small datasets, is easy to train locally, and does not require external APIs. Most importantly, it supports **explainable decision-making**, which is critical for educational applications.

### 3. Explainability Over Complexity

Rather than using large language models, the system focuses on transparency. Each recommendation is accompanied by a clear explanation (e.g., interest in technology, budget preference, early employment goal), making the system trustworthy and user-friendly.

### 4. Reliability and Stability

The system avoids third-party API dependencies to ensure consistent performance during evaluation and demonstration. This design choice improves reliability and simplifies deployment.

---

## Improvements With More Time

With additional time and resources, the following improvements would be implemented:

* Collection of real-world student data to improve model accuracy
* Expansion of course and college-level recommendations
* Multilingual support (Malayalam and English)
* Parent and counselor dashboards
* Integration of admission cutoffs and employability trends
* Use of ensemble models or hybrid AI approaches for better predictions

---

## Conclusion

This project demonstrates how AI can be applied responsibly in education by prioritizing explainability, reliability, and user-centric design. The system serves as a strong foundation that can be scaled into a full-fledged career guidance platform with real-world impact.

**Architecture Explanation – Career Guidance AI**

## System Overview

The Career Guidance AI system is designed as a **lightweight, modular, offline application** that provides career recommendations using a trained machine learning model. The architecture prioritizes **simplicity, explainability, and reliability**.

---

## 1. System Components

### 1. User Interface (Streamlit)

* Collects student inputs through a web form
* Displays recommended courses and explanations
* Acts as the interaction layer between the user and the system

### 2. Feature Mapping Layer

* Converts qualitative user inputs (interests, goals, constraints) into numerical features
* Ensures compatibility with the machine learning model

### 3. AI / ML Engine

* Uses a locally trained **Decision Tree model**
* Predicts the most suitable course based on input features
* Generates human-readable explanations for recommendations

### 4. Data Storage Layer

* Stores user responses locally in JSON format
* Stores training data in CSV format
* Stores trained model as a serialized file

---

## 2. API Flow

This system does **not depend on external APIs**.
All processing happens locally.

### Flow:

1. User submits form in the UI
2. Input data is mapped to numerical features
3. Features are passed to the ML model
4. Model returns a predicted course
5. Explanation logic generates reasoning
6. Result is displayed to the user

This approach avoids network dependency and quota limitations.

---

## 3. AI Integration Approach

* The AI component is implemented using a **Decision Tree Classifier**
* The model is trained offline using structured data
* During runtime:

  * The model is loaded into memory
  * Predictions are made synchronously
* Explainability is handled using rule-based interpretation of input features

This approach ensures:

* Transparency
* Predictable behavior
* Easy debugging and enhancement

---

## 4. Data Storage Approach

### Training Data

* Stored in `data/training_data.csv`
* Contains structured features and corresponding course labels

### Model Storage

* Trained model saved as `career_model.pkl` using `joblib`

### User Responses

* Stored in `data/responses.json`
* Used for logging and future analysis (not for training in current version)

All data is stored **locally**, making the system portable and secure.

**AI Development Workflow**

## AI Tools Used

During development, AI tools were used primarily as **assistance tools**, not as core system dependencies.

* Large Language Models (LLMs) were used during early ideation and prototyping to:

  * Refine problem understanding
  * Draft initial question sets
  * Validate design approaches
* No external AI APIs are used in the final running system

The final application uses a **locally trained machine learning model** instead of live AI services.

---

## AI-Assisted Parts

AI assistance was used in the following areas:

* Brainstorming the problem statement and solution approach
* Structuring career guidance questions logically
* Designing feature mappings from qualitative inputs to numerical values
* Reviewing and refining system architecture and documentation

These tools acted as **guides and reviewers**, not automated code generators.

---

## Self-Implemented Components

The following components were fully implemented and controlled by the developer:

* Dataset design and feature selection
* Model training using a Decision Tree classifier
* Model evaluation and tuning
* Explainability logic for recommendations
* Streamlit user interface
* Local data storage and model persistence
* Dependency management and environment setup

All final code, model logic, and integration decisions were manually implemented to ensure reliability, transparency, and correctness.

---

## Summary

AI tools were used responsibly to support design thinking and development efficiency, while all critical system logic, model training, and deployment decisions were implemented independently. This approach ensured a stable, explainable, and production-ready solution.




**Edge Cases and Limitations**

## 1. External AI API Instability (During Development)

During early stages of development, multiple external AI APIs were evaluated (OpenAI, Google Gemini, Hugging Face). These integrations faced several issues:

* API quota exhaustion and rate limiting
* Model deprecation or unavailability
* Version incompatibilities in SDKs
* Authentication and environment variable errors

These issues made the system unreliable for consistent demonstration and evaluation. As a result, the final design intentionally avoids live API dependencies and uses a locally trained model instead.

---

## 2. Limited Dataset Size

The machine learning model is trained on a **small, synthetic dataset** created for demonstration purposes.
Limitations include:

* Reduced prediction accuracy
* Limited ability to generalize to all student profiles
* Potential bias toward predefined patterns

In a real-world deployment, this would be addressed by collecting a large and diverse dataset from actual students.

---

## 3. Simplified Feature Representation

Complex human attributes such as interests, personality, and motivation are simplified into binary or categorical values.
This may not fully capture:

* Mixed interests
* Changing career goals
* Nuanced personal preferences

The simplification was a conscious design choice to keep the system explainable and manageable.

---

## 4. No Real-Time Educational Data

The system does not currently include:

* College admission cutoffs
* Course availability updates
* Job market trends
* Government policy changes

As a result, recommendations are **guidance-oriented**, not contextually dynamic.

---

## 5. User Input Quality Dependency

The system assumes that users provide:

* Honest answers
* Reasonably accurate self-assessments

Incorrect or random inputs can lead to poor recommendations, as no validation or behavioral analysis is currently implemented.

---

## 6. Limited Scope of Recommendations

The current model focuses on **broad course-level recommendations**.
It does not:

* Rank colleges
* Suggest specific institutions
* Provide personalized career roadmaps

This limits its usefulness for advanced planning but keeps the system simple and reliable.

---

## 7. Not a Replacement for Human Counseling

While the system provides structured guidance, it cannot:

* Understand emotional or social pressures
* Replace professional career counselors
* Account for family or cultural expectations

It is designed as a **decision-support tool**, not a definitive authority.
